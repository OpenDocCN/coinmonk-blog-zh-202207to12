去中心化身份( DID )是Web3的主要特性之一，与Web2用户的区别在于所有权和控制权属于用户本身。

本文将展示如何实现基于Web3身份的认证和授权程序，以及如何将其与 Springboot框架集成

Web3身份

Web3身份由一个只有所有者知道的私钥和一个任何人都知道的公钥组成。

私钥被所有者用来签署数据——例如在区块链的交易——公钥被签名的接收者用来验证数据。

Web2身份的不同之处在于，你不需要第三方来证明你的身份——例如脸书、谷歌、Okta等。—使用您和身份管理员( IAM )都知道的用户名/密码

在Web3中，密码用于锁定或加密存储私钥的钱包，而不是用于身份验证。

身份证明

Web3中的用户通过用他们的私钥生成的签名来证明他们的身份。

  • 应用程序可以要求用户通过请求来证明他们的身份，这意味着用户必须通过请求生成签名。一个例子是基于区块链网络的 Dapps 。
  • 应用程序可以要求用户只需证明一次身份就能获得访问权限，例如传统的登录— 。在这种情况下，用户对任意文本进行签名，并将该签名用于发送到应用程序的所有请求。

授权服务器

授权服务器也称为 IAM ，是一个管理给定应用程序中用户的认证和授权的系统。

  • 认证是验证用户身份的过程。例如，在基于用户名/密码的系统中，通过提供密码来证明用户名的身份。
  • 授权是为给定的已认证身份确定授予权限的过程。例如授予给定用户的权限或角色。

授权服务器的实现

授权服务器的提出是受 OAuth 协议的启发

用户在授权服务器上证明他们的身份，授权服务器用用户的身份和权限导出一个 JWT 。应用程序使用 JWT 来授权访问私有资源。

与标准的 OAuth 服务器不同，Web3授权服务器不提供登录页面，也不存储任何用户数据。登录页面被用户的钱包所取代——例如元掩码——授权来源于NFT所有权。

让我们看看它是如何工作的:

  1. 用户试图登录应用程序。显示的只是一个按钮，没有用户名和密码字段。

2.用户点击按钮，并被要求用元掩码签署一个消息。

3.签名被发送到授权服务器，授权服务器验证签名并重定向到回调url，产生 JWT 

http://localhost:9080/authorize?token=eyJ0eXAiO...H6BD1g23-A

JSON Web令牌的模式

由 JWT 提供的有效载荷中的数据是:

用户的主题 : 地址或公钥

范围:用户拥有的NFT的名称。

exp :令牌的到期日期

与Springboot框架集成

Springboot与Web3身份的集成在于 Spring安全项目

  • Spring security filters 可用于从报头获取 JWT 或签名。
  • Spring安全认证提供者可用于验证签名，并通过授权映射NFT所有权。
